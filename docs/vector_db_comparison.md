# 向量 DB 選型筆記

## 測試環境

- 資料量：20 個文字段落（來自 3 個品牌的 11 份知識庫文件）
- Embedding 模型：`all-MiniLM-L6-v2`（384 維，本機執行）
- 測試查詢：3 個繁體中文語意搜尋問句

## 實驗結果

### 寫入速度

| DB | 耗時 | 備註 |
|--|--|--|
| pgvector | 0.01s | 直接寫本機 PostgreSQL |
| ChromaDB | 0.29s | 本機，有 Python overhead |
| Pinecone | 3.29s | 透過網路上傳至雲端 |

### 查詢速度（3 個問句平均）

| DB | 耗時 | 備註 |
|--|--|--|
| pgvector | 2.9ms | 本機，無網路延遲 |
| ChromaDB | 37.8ms | 本機 |
| Pinecone | 303.6ms | 網路來回延遲 |

### 查詢結果一致性

三個 DB 在所有查詢上結果**完全一致**，驗證了向量搜尋的正確性與可重現性。

## 關鍵觀察：Embedding 的局限性

實驗中發現：查詢「山嵐咖啡線上銷售下滑的原因」時，三個 DB 都把`鮮橙生活_退貨率異常`排到第一名（相似度 0.65），正確的山嵐咖啡文件反而排第三（0.53）。

**原因**：兩份文件的文字風格太相近（都在談退款、修正流程），embedding 模型抓的是語意風格而非品牌名稱。

**解法**：pgvector 可以加 SQL 條件先過濾品牌，再做向量搜尋，精準度大幅提升：

```sql
SELECT text FROM documents
WHERE brand = '山嵐咖啡'
ORDER BY embedding <=> %s
LIMIT 3;
```

ChromaDB 和 Pinecone 做不到這種先過濾再搜尋的組合查詢。

## 各 DB 特性比較

| | ChromaDB | Pinecone | pgvector |
|--|--|--|--|
| 部署方式 | 本機檔案 | 雲端託管 | Docker / 自建 |
| 需要網路 | 否 | 是 | 否 |
| 費用 | 免費 | 免費額度 / 付費 | 免費 |
| 混合 SQL 查詢 | 否 | 否 | 是 |
| 維運成本 | 低 | 零（託管） | 中（需維護 PostgreSQL） |
| 適合資料量 | < 10 萬筆 | 百萬筆以上 | 視索引設定 |

## 選型建議

**開發 / 原型階段 → ChromaDB**

零設定，`pip install chromadb` 即可使用，適合快速驗證 RAG 流程。

**生產環境，需要高可用 → Pinecone**

完全託管，不需要維護伺服器。適合團隊沒有 infra 資源，或需要快速上線的場景。缺點是資料存在第三方雲端，有資料外流疑慮。

**已有 PostgreSQL 基礎設施 → pgvector**

向量和結構化資料存在同一個資料庫，可以用 SQL 條件精確控制搜尋範圍。適合行銷分析這類「需要先過濾品牌 / 時間範圍，再做語意搜尋」的場景。

**本專案選擇**：開發階段用 ChromaDB，若要上生產環境，優先考慮 pgvector（因為行銷數據已有結構化的品牌、時間欄位，混合查詢優勢明顯）。
